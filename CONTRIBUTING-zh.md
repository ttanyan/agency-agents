## 📜 行为准则

本项目和参与其中的每个人都受我们的行为准则约束。通过参与，您应该遵守此准则：

- **尊重他人**：尊重对待每个人。我们鼓励健康的辩论，但不容忍人身攻击。
- **包容性**：欢迎和支持所有背景和身份的人。
- **协作**：我们一起创造的东西比单独创造的更好。
- **专业**：保持讨论集中在改进代理和社区上。

## 🎯 如何贡献？

### 1. 创建新代理

有新的专业代理想法吗？太好了！以下是添加方法：

1. **Fork 仓库**
2. **选择适当的类别**（或提议新的类别）：
   - `engineering/` - 软件开发专家
   - `design/` - UX/UI 和创意专家
   - `marketing/` - 增长和营销专家
   - `product/` - 产品管理专家
   - `project-management/` - 项目管理和协调专家
   - `testing/` - QA 和测试专家
   - `support/` - 运营和支持专家
   - `spatial-computing/` - AR/VR/XR 专家
   - `specialized/` - 不适合其他地方的独特专家

3. **按照以下模板创建您的代理文件**
4. **在真实场景中测试您的代理**
5. **提交 Pull Request** 并附上您的代理

### 2. 改进现有代理

找到了让代理更好的方法？欢迎贡献：

- 添加现实世界的示例和用例
- 用现代模式增强代码示例
- 根据新的最佳实践更新工作流程
- 添加成功指标和基准
- 修复拼写错误、提高清晰度、增强文档

### 3. 分享成功故事

成功使用这些代理了吗？分享您的故事：

- 在 [GitHub Discussions](https://github.com/msitarzewski/agency-agents/discussions) 中发帖
- 在 README 中添加案例研究
- 撰写博客文章并链接
- 创建视频教程

### 4. 报告问题

发现了问题？告诉我们：

- 检查问题是否已经存在
- 提供清晰的重现步骤
- 包含有关您用例的上下文
- 如果有想法，建议潜在的解决方案

## 🎨 代理设计指南

### 代理文件结构

每个代理都应遵循以下结构：

```markdown
---
name: 代理名称
description: 代理专业领域和焦点的一行描述
color: 颜色名称或 "#hex 代码"
---

# 代理名称

## 🧠 您的身份和记忆
- **角色**：清晰的角色描述
- **个性**：个性特征和沟通风格
- **记忆**：代理记住和学习的内容
- **经验**：领域专业知识和视角

## 🎯 您的核心使命
- 带有清晰交付物的主要职责 1
- 带有清晰交付物的主要职责 2
- 带有清晰交付物的主要职责 3
- **默认要求**：始终开启的最佳实践

## 🚨 您必须遵守的关键规则
定义代理方法的特定领域规则和约束

## 📋 您的技术交付物
代理生成的具体示例：
- 代码示例
- 模板
- 框架
- 文档

## 🔄 您的工作流程
代理遵循的逐步流程：
1. 阶段 1：发现和研究
2. 阶段 2：规划和策略
3. 阶段 3：执行和实施
4. 阶段 4：审查和优化

## 💭 您的沟通风格
- 代理如何沟通
- 示例短语和模式
- 语气和方法

## 🔄 学习和记忆
代理学习的内容：
- 成功模式
- 失败方法
- 用户反馈
- 领域演进

## 🎯 您的成功指标
可衡量的结果：
- 定量指标（带数字）
- 定性指标
- 性能基准

## 🚀 高级能力
代理掌握的高级技术和方法
```

### 代理设计原则

1. **🎭 强烈的个性**
   - 赋予代理独特的声音和角色
   - 不是"我是一个有用的助手" - 要具体和令人难忘
   - 示例："我默认会发现 3-5 个问题，并且需要所有事情的视觉证明"（证据收集器）

2. **📋 清晰的交付物**
   - 提供具体的代码示例
   - 包括模板和框架
   - 显示真实的输出，不是模糊的描述

3. **✅ 成功指标**
   - 包括具体的、可衡量的指标
   - 示例："页面加载时间在 3G 网络下低于 3 秒"
   - 示例："跨账户累计 10,000+ karma"

4. **🔄 经验证的工作流程**
   - 逐步流程
   - 现实世界测试过的方法
   - 不是理论上的 - 是经过实战检验的

5. **💡 学习记忆**
   - 代理识别的模式
   - 如何随时间改进
   - 在会话之间记住的内容

### 什么是优秀的代理？

**优秀的代理有**：
- ✅ 狭窄、深入的专业化
- ✅ 独特的个性和声音
- ✅ 具体的代码/模板示例
- ✅ 可衡量的成功指标
- ✅ 逐步工作流程
- ✅ 现实世界的测试和迭代

**避免**：
- ❌ 通用的"有帮助的助手"个性
- ❌ 模糊的"我会帮助你..."描述
- ❌ 没有代码示例或交付物
- ❌ 范围过于广泛（万事通）
- ❌ 未经测试的理论方法

## 🔄 Pull Request 流程

### 提交之前

1. **测试您的代理**：在真实场景中使用它，根据反馈迭代
2. **遵循模板**：匹配现有代理的结构
3. **添加示例**：包括至少 2-3 个代码/模板示例
4. **定义指标**：包括具体的、可衡量的成功标准
5. **校对**：检查拼写错误、格式问题、清晰度

### 提交您的 PR

1. **Fork** 仓库
2. **创建分支**: `git checkout -b add-agent-name`
3. **进行更改**：添加您的代理文件
4. **提交**: `git commit -m "Add [Agent Name] specialist"`
5. **推送**: `git push origin add-agent-name`
6. **打开 Pull Request** 包含：
   - 清晰的标题："Add [Agent Name] - [Category]"
   - 代理作用的描述
   - 为什么需要这个代理（用例）
   - 您进行的任何测试

### PR 审查流程

1. **社区审查**：其他贡献者可能提供反馈
2. **迭代**：解决反馈并进行改进
3. **批准**：维护者会在准备好时批准
4. **合并**：您的贡献成为 The Agency 的一部分！

### PR 模板

```markdown
## 代理信息
**代理名称**: [名称]
**类别**: [engineering/design/marketing 等]
**专业领域**: [一行描述]

## 动机
[为什么需要这个代理？它填补了什么空白？]

## 测试
[您如何测试这个代理？现实世界的用例？]

## 清单
- [ ] 遵循代理模板结构
- [ ] 包含个性和声音
- [ ] 有具体的代码/模板示例
- [ ] 定义了成功指标
- [ ] 包含逐步工作流程
- [ ] 已校对并正确格式化
- [ ] 在真实场景中测试过
```

## 📐 风格指南

### 写作风格

- **具体**："减少页面加载时间 60%"而不是"让它更快"
- **明确**："用 TypeScript 创建 React 组件"而不是"构建 UI"
- **令人难忘**：赋予代理个性，不是通用的企业用语
- **实用**：包含真实的代码，不是伪代码

### 格式

- 一致地使用 **Markdown 格式**
- 为章节标题包含 **emoji**（使扫描更容易）
- 为所有代码示例使用 **代码块**，并带有适当的语法高亮
- 使用 **表格** 比较选项或显示指标
- 使用 **粗体** 强调，`code` 表示技术术语

### 代码示例

```markdown
## 示例代码块

\`\`\`typescript
// 始终包括：
// 1. 语言规范用于语法高亮
// 2. 解释关键概念的注释
// 3. 真实的、可运行的代码（不是伪代码）
// 4. 现代最佳实践

interface AgentExample {
  name: string;
  specialty: string;
  deliverables: string[];
}
\`\`\`
```

### 语气

- **专业但平易近人**：不过于正式或随意
- **自信但不傲慢**："这是最好的方法"而不是"也许你可以试试..."
- **有帮助但不手把手教导**：假设有能力，提供深度
- **个性驱动**：每个代理都应该有独特的声音

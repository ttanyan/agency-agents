# 多代理工作流：初创公司 MVP

> 一个逐步示例，展示如何协调多个代理从想法到发布的 MVP。

## 场景

您正在构建 SaaS MVP -- 一个面向远程团队的团队回顾工具。您有 4 周时间发布具有用户注册、核心功能和着陆页的工作产品。

## 代理团队

| 代理 | 在此工作流中的角色 |
|-------|---------------------|
| 冲刺优先级排序器 | 将项目分解为每周冲刺 |
| UX 研究员 | 通过快速用户访谈验证想法 |
| 后端架构师 | 设计 API 和数据模型 |
| 前端开发者 | 构建 React 应用 |
| 快速原型师 | 让第一版快速运行 |
| 增长黑客 | 在构建时规划发布策略 |
| 现实检验者 | 在每个里程碑之前检查才能继续 |

## 工作流程

### 第 1 周：发现 + 架构

**步骤 1 — 激活冲刺优先级排序器**

```
激活冲刺优先级排序器。

项目：RetroBoard -- 面向远程团队的实时团队回顾工具。
时间线：4 周到 MVP 发布。
核心功能：用户认证、创建回顾板、添加卡片、投票、行动项目。
约束：独立开发者，React + Node.js 技术栈，部署到 Vercel + Railway。

将此分解为 4 个每周冲刺，具有清晰的交付成果和验收标准。
```

**步骤 2 — 激活 UX 研究员（并行）**

```
激活 UX 研究员。

我正在为远程团队（5-20 人）构建团队回顾工具。
竞争对手：EasyRetro、Retrium、Parabol。

进行快速竞争分析并识别：
1. 什么是必备功能
2. 竞争对手的不足之处
3. 我们可以拥有的一个差异化优势

输出 1 页研究简报。
```

**步骤 3 — 移交给后端架构师**

```
激活后端架构师。

这是我们的冲刺计划：[粘贴冲刺优先级排序器输出]
这是我们的研究简报：[粘贴 UX 研究员输出]

设计 RetroBoard 的 API 和数据库 schema。
技术栈：Node.js、Express、PostgreSQL、Socket.io 用于实时。

交付：
1. 数据库 schema（SQL）
2. REST API 端点列表
3. 用于实时更新板的 WebSocket 事件
4. 认证策略建议
```

### 第 2 周：构建核心功能

**步骤 4 — 激活前端开发者 + 快速原型师**

```
激活前端开发者。

这是 API 规格：[粘贴后端架构师输出]

构建 RetroBoard React 应用：
- 技术栈：React、TypeScript、Tailwind、Socket.io-client
- 页面：登录、仪表板、板视图
- 组件：RetroCard、VoteButton、ActionItem、BoardColumn

从板视图开始 -- 这是核心体验。
专注于实时：当一个用户添加卡片时，每个人都能看到。
```

**步骤 5 — 中途现实检验**

```
激活现实检验者。

我们在 4 周 MVP 构建的第 2 周。

这是我们目前拥有的：
- 数据库 schema：[粘贴]
- API 端点：[粘贴]
- 前端组件：[粘贴]

评估：
1. 我们是否能在 2 周内实际发布？
2. 我们应该削减什么以满足截止日期？
3. 任何会在发布时困扰我们的技术债务？
```

### 第 3 周：优化 + 着陆页

**步骤 6 — 前端开发者继续，增长黑客开始**

```
激活增长黑客。

产品：RetroBoard -- 团队回顾工具，1 周后发布。
目标：远程优先公司的工程经理和 scrum master。
预算：0 美元（仅自然发布）。

创建发布计划：
1. 着陆页文案（hero、功能、CTA）
2. 发布渠道（Product Hunt、Reddit、Hacker News、Twitter）
3. 逐日发布序列
4. 第 1 周要追踪的指标
```

### 第 4 周：发布

**步骤 7 — 最终现实检验**

```
激活现实检验者。

RetroBoard 准备发布。评估生产就绪性：

- 实时 URL：[url]
- 测试账户已创建：是
- 错误监控：Sentry 已配置
- 数据库备份：每日自动化

完成发布清单并给出 GO / NO-GO 决策。
需要每个标准的证据。
```

## 关键模式

1. **顺序移交**：每个代理的输出成为下一个代理的输入
2. **并行工作**：UX 研究员和冲刺优先级排序器可以在第 1 周同时运行
3. **质量门**：中点和发布前的现实检验者防止发布破碎的代码
4. **上下文传递**：始终将先前的代理输出粘贴到下一个提示中 -- 代理不共享记忆

## 提示

- 在步骤之间复制粘贴代理输出 -- 不要总结，使用完整输出
- 如果现实检验者标记了问题，返回给相关专家修复
- 一旦您熟悉了手动版本，请记住编排器代理可以自动化此流程

## 1. 战略基础

### 1.1 NEXUS 解决的问题

单个代理很强大。但没有协调，它们会产生：
- 冲突的架构决策
- 跨部门的重复工作
- 交接边界的质量差距
- 没有共享的上下文或机构记忆

**NEXUS 消除这些失败模式**，通过定义：
- **谁**在每个阶段激活
- **什么**他们生产以及为谁生产
- **何时**他们交接以及交接给谁
- **如何**在前进之前验证质量
- **为什么**每个代理存在于管道中（没有乘客）

### 1.2 核心原则

| 原则 | 描述 |
|-----------|-------------|
| **管道完整性** | 没有阶段在通过其质量门之前前进 |
| **上下文连续性** | 每个交接都携带完整的上下文 — 没有代理冷启动 |
| **并行执行** | 独立工作流同时运行以压缩时间线 |
| **证据胜于声明** | 所有质量评估都需要证明，而不是断言 |
| **快速失败，快速修复** | 每个任务最多3次重试，然后升级 |
| **单一事实来源** | 一个规范，一个任务列表，一个架构文档 |

| 部门 | 代理 | 主要NEXUS角色 |
|----------|--------|--------------------|

| **设计** | UI设计师、UX研究员、UX架构师、品牌守护者、视觉讲述者、奇思妙想注入者、图像提示工程师 | 定义视觉标识、用户体验和品牌一致性 |

```
│                                                                         │
```

```
                    ┌──────────────────────┐

                    └──────────┬───────────┘
```

### 2.3 激活模式

NEXUS 支持三种部署配置：

| 模式 | 活跃代理 | 用例 | 时间线 |
|------|--------------|----------|----------|
| **NEXUS-Full** | 全部 | 企业产品启动，完整生命周期 | 12-24周 |
| **NEXUS-Sprint** | 15-25 | 功能开发，MVP构建 | 2-6周 |
| **NEXUS-Micro** | 5-10 | 错误修复，内容活动，单一交付物 | 1-5天 |

## 3. 第0阶段 — 情报与发现

> **目标**：在投入资源之前了解格局。在问题得到验证之前不进行构建。

### 3.1 活跃代理

| 代理 | 阶段中的角色 | 主要输出 |
|-------|--------------|----------------|
| **趋势研究员** | 市场情报负责人 | 带有TAM/SAM/SOM的市场分析报告 |
| **反馈合成器** | 用户需求分析 | 带有痛点的合成反馈报告 |
| **UX研究员** | 用户行为分析 | 带有角色和旅程地图的研究发现 |
| **分析报告员** | 数据格局评估 | 带有可用信号的数据审计报告 |
| **法律合规检查器** | 监管扫描 | 合规要求矩阵 |
| **工具评估师** | 技术格局 | 技术栈评估 |

### 3.2 并行工作流

```
工作流A：市场情报          工作流B：用户情报
├── 趋势研究员                       ├── 反馈合成器
│   ├── 竞争格局              │   ├── 多渠道反馈收集
│   ├── 市场规模（TAM/SAM/SOM）        │   ├── 情感分析
│   └── 趋势生命周期映射            │   └── 痛点优先级
│                                          │
├── 分析报告员                     ├── UX研究员
│   ├── 现有数据审计                │   ├── 用户访谈/调查
│   ├── 信号识别              │   ├── 角色开发
│   └── 基线指标                   │   └── 旅程映射
│                                          │
└── 法律合规检查器               └── 工具评估师
    ├── 监管要求                ├── 技术评估
    ├── 数据处理约束               ├── 构建vs购买分析
    └── 司法管辖区映射                   └── 集成可行性
```

### 3.3 第0阶段质量门

**守门员**：执行摘要生成器

| 标准 | 阈值 | 所需证据 |
|-----------|-----------|-------------------|
| 市场机会已验证 | TAM > 最小可行阈值 | 带有来源的趋势研究员报告 |
| 用户需求已确认 | ≥3个已验证的痛点 | 反馈合成器 + UX研究员数据 |
| 监管路径清晰 | 无阻塞性合规问题 | 法律合规检查器矩阵 |
| 数据基础已评估 | 关键指标已识别 | 分析报告员审计 |
| 技术可行性已确认 | 栈已验证 | 工具评估师评估 |

**输出**：执行摘要（≤500字，SCQA格式）→ 决策：GO / NO-GO / PIVOT

## 4. 第1阶段 — 策略与架构

> **目标**：在编写一行代码之前，定义我们正在构建什么、它的结构如何以及成功是什么样子。

### 4.1 活跃代理

| 代理 | 阶段中的角色 | 主要输出 |
|-------|--------------|----------------|
| **工作室制作人** | 战略组合对齐 | 战略组合计划 |
| **高级项目经理** | 规范到任务转换 | 综合任务列表 |
| **Sprint优先级排序器** | 功能优先级 | 优先级待办事项（RICE评分） |
| **UX架构师** | 技术架构 + UX基础 | 架构规范 + CSS设计系统 |
| **品牌守护者** | 品牌标识系统 | 品牌基础文档 |
| **后端架构师** | 系统架构 | 系统架构规范 |
| **AI工程师** | AI/ML架构（如适用） | ML系统设计 |
| **财务跟踪器** | 预算和资源规划 | 带有ROI预测的财务计划 |

### 4.2 执行序列

```
步骤1：战略框架（并行）
├── 工作室制作人 → 战略组合计划（愿景、目标、ROI目标）
├── 品牌守护者 → 品牌基础（目的、价值观、视觉标识系统）
└── 财务跟踪器 → 预算框架（资源分配、成本预测）

步骤2：技术架构（并行，步骤1之后）
├── UX架构师 → CSS设计系统 + 布局框架 + UX结构
├── 后端架构师 → 系统架构（服务、数据库、API）
├── AI工程师 → ML架构（模型、管道、推理策略）
└── 高级项目经理 → 任务列表（规范 → 任务，确切要求）

步骤3：优先级排序（顺序，步骤2之后）
└── Sprint优先级排序器 → RICE评分的待办事项，带有sprint分配
    ├── 输入：任务列表 + 架构规范 + 预算框架
    ├── 输出：带有依赖映射的优先级sprint计划
    └── 验证：工作室制作人确认战略对齐
```

### 4.3 第1阶段质量门

**守门员**：工作室制作人 + 现实检查器（双重签字）

| 标准 | 阈值 | 所需证据 |
|-----------|-----------|-------------------|
| 架构覆盖所有要求 | 100%规范覆盖 | 高级PM任务列表交叉引用 |
| 品牌系统完整 | 标志、颜色、排版、声音已定义 | 品牌守护者交付物 |
| 技术可行性已验证 | 所有组件都有实现路径 | 后端架构师 + UX架构师规范 |
| 预算已批准 | 在组织约束范围内 | 财务跟踪器计划 |
| Sprint计划现实 | 基于速度的估计 | Sprint优先级排序器待办事项 |

**输出**：已批准的架构包 → 第2阶段激活

## 6. 第3阶段 — 构建与迭代

### 6.1 Dev↔QA 循环

```

│                                                          │
│  ┌──────────┐    ┌──────────┐    ┌──────────────────┐   │

│  │ 任务 N    │    │ 测试    │    │ BLOCKED → 升级│   │
│  │           │◀───│ 任务 N   │◀───│                   │   │
│  └──────────┘    └──────────┘    └──────────────────┘   │
│       ▲                                    │             │
│       │            QA 反馈             │             │
│       └────────────────────────────────────┘             │
│                                                          │
│  编排器跟踪：尝试次数，QA反馈，        │
```

## 附录C：NEXUS词汇表

| 术语 | 定义 |
|------|-----------|
| **NEXUS** | 专家网络，战略统一 |
| **质量门** | 阶段之间的强制检查点，需要基于证据的批准 |
| **Dev↔QA循环** | 持续开发-测试周期，每个任务必须通过QA才能继续 |
| **交接** | 代理之间结构化的工作和上下文转移 |
| **守门员** | 有权批准或拒绝阶段前进的代理 |
| **升级** | 在重试用尽后将阻塞的任务路由到更高权限 |
| **NEXUS-Full** | 所有代理的完整管道激活 |
| **NEXUS-Sprint** | 15-25个代理的聚焦管道，用于功能/MVP工作 |
| **NEXUS-Micro** | 5-10个代理的目标激活，用于特定任务 |
| **管道完整性** | 没有质量门通过，阶段不能前进的原则 |
| **上下文连续性** | 每个交接都携带完整上下文的原则 |
| **证据胜于声明** | 质量评估需要证明而不是断言的原则 |

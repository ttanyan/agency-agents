# 🔨 第 3 阶段手册 — 构建与迭代

> **持续时间**：2-12 周（根据范围而异） | **代理**：15-30+ | **门控守护者**：代理编排器

---

## 目标

通过持续的 Dev↔QA 循环实现所有功能。每个任务在下一个任务开始前都经过验证。这是大部分工作发生的地方 — 也是 NEXUS 编排提供最大价值的地方。

## 前提条件

- [ ] 第 2 阶段质量门控通过（基础已验证）
- [ ] 带有 RICE 评分的冲刺优先排序器待办事项可用
- [ ] CI/CD 管道运行中
- [ ] 设计系统和组件库就绪
- [ ] 带有认证系统的 API 脚手架就绪

## Dev↔QA 循环 — 核心机制

代理编排器通过以下循环管理每个任务：

```
对于 sprint_backlog 中的每个任务（按 RICE 评分排序）：

  1. 将任务分配给适当的开发者代理（见分配矩阵）
  2. 开发者实现任务
  3. 证据收集器测试任务
     - 视觉截图（桌面、平板、移动）
     - 根据验收标准进行功能验证
     - 品牌一致性检查
  4. 如果 verdict == 通过：
       标记任务完成
       移动到下一个任务
     否则如果 verdict == 失败且尝试 < 3：
       向开发者发送 QA 反馈
       开发者修复特定问题
       返回步骤 3
     否则如果尝试 >= 3：
       升级到代理编排器
       编排器决定：重新分配、分解、延迟或接受
  5. 更新管道状态报告
```

## 并行构建轨道

对于 NEXUS-Full 部署，四个轨道同时运行：

### 轨道 A：核心产品开发
```
管理：代理编排器（Dev↔QA 循环）
代理：前端开发者、后端架构师、AI 工程师、
        移动应用构建者、高级开发者
QA：证据收集器、API 测试器、测试结果分析器

冲刺节奏：2 周冲刺
日常：任务实现 + QA 验证
冲刺结束：冲刺评审 + 回顾
```

### 轨道 B：增长与营销准备
```
管理：项目牧羊人
代理：增长黑客、内容创建者、社交媒体策略师、
        应用商店优化师

冲刺节奏：与轨道 A 里程碑对齐
活动：
- 增长黑客 → 设计病毒循环和推荐机制
- 内容创建者 → 构建发布内容管道
- 社交媒体策略师 → 计划跨平台活动
- 应用商店优化师 → 准备商店列表（如果是移动应用）
```

### 轨道 C：质量与运营
```
管理：代理编排器
代理：证据收集器、API 测试器、性能基准测试器、
        工作流优化器、实验跟踪器

持续活动：
- 证据收集器 → 每个任务的截图 QA
- API 测试器 → 每个 API 任务的端点验证
- 性能基准测试器 → 定期负载测试
- 工作流优化器 → 流程改进识别
- 实验跟踪器 → 已验证功能的 A/B 测试设置
```

### 轨道 D：品牌与体验润色
```
管理：品牌守护者
代理：UI 设计师、品牌守护者、视觉讲故事者、
        奇思妙想注入器

触发活动：
- UI 设计师 → 当 QA 识别到视觉问题时进行组件改进
- 品牌守护者 → 定期品牌一致性审计
- 视觉讲故事者 → 随着功能完成提供视觉叙事资产
- 奇思妙想注入器 → 微交互和愉悦时刻
```

## 冲刺执行模板

### 冲刺规划（第 1 天）

```
冲刺优先排序器激活：
1. 查看带有更新 RICE 评分的待办事项
2. 根据团队速度选择冲刺任务
3. 将任务分配给开发者代理
4. 识别依赖关系和排序
5. 设置冲刺目标和成功标准

输出：带有任务分配的冲刺计划
```

### 日常执行（第 2 天至第 N-1 天）

```
代理编排器管理：
1. 当前任务状态检查
2. Dev↔QA 循环执行
3. 阻塞识别和解决
4. 进度跟踪和报告

状态报告格式：
- 今天完成的任务：[列表]
- 在 QA 中的任务：[列表]
- 在开发中的任务：[列表]
- 阻塞的任务：[带原因的列表]
- QA 通过率：[X/Y]
```

### 冲刺评审（第 N 天）

```
项目牧羊人主持：
1. 演示已完成的功能
2. 审查每个任务的 QA 证据
3. 收集利益相关者反馈
4. 根据学习更新待办事项

参与者：所有活跃代理 + 利益相关者
输出：冲刺评审摘要
```

### 冲刺回顾

```
工作流优化器主持：
1. 什么做得好？
2. 什么可以改进？
3. 下一个冲刺我们将改变什么？
4. 流程效率指标

输出：回顾行动项
```

## 编排器决策逻辑

### 任务失败处理

```
当任务失败 QA 时：
  如果尝试 == 1：
    → 向开发者发送具体的 QA 反馈
    → 开发者仅修复已识别的问题
    → 重新提交 QA
    
  如果尝试 == 2：
    → 发送累积的 QA 反馈
    → 考虑：开发者代理是否合适？
    → 开发者在额外上下文下修复
    → 重新提交 QA
    
  如果尝试 == 3：
    → 升级
    → 选项：
      a) 重新分配给不同的开发者代理
      b) 将任务分解为更小的子任务
      c) 修改方法/架构
      d) 接受已知限制（记录）
      e) 延迟到未来冲刺
    → 记录决策和理由
```

### 并行任务管理

```
当多个任务没有依赖关系时：
  → 同时分配给不同的开发者代理
  → 每个运行独立的 Dev↔QA 循环
  → 编排器同时跟踪所有循环
  → 按依赖顺序合并已完成的任务

当任务有依赖关系时：
  → 等待依赖项通过 QA
  → 然后分配依赖任务
  → 在交接中包含依赖上下文
```

## 质量门控清单

| # | 标准 | 证据来源 | 状态 |
|---|-------|----------|------|
| 1 | 所有冲刺任务通过 QA（100% 完成） | 证据收集器每个任务的截图 | ☐ |
| 2 | 所有 API 端点已验证 | API 测试器回归报告 | ☐ |
| 3 | 性能基线满足（P95 < 200ms） | 性能基准测试器报告 | ☐ |
| 4 | 品牌一致性已验证（95%+ 遵守） | 品牌守护者审计 | ☐ |
| 5 | 无关键错误（零 P0/P1 开放） | 测试结果分析器摘要 | ☐ |
| 6 | 所有验收标准满足 | 逐个任务验证 | ☐ |
| 7 | 所有 PR 的代码审查已完成 | Git 历史证据 | ☐ |

## 门控决策

**门控守护者**：代理编排器

- **通过**：功能完整的应用 → 激活第 4 阶段
- **继续**：需要更多冲刺 → 继续第 3 阶段
- **升级**：系统性问题 → 工作室制作人干预

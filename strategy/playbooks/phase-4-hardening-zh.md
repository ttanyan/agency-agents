# 🛡️ 第 4 阶段手册 — 质量与加固

> **持续时间**：3-7 天 | **代理**：8 | **门控守护者**：现实检查器（唯一权威）

---

## 目标

最后的质量关卡。现实检查器默认为"需要工作" — 你必须用压倒性的证据证明生产就绪性。这个阶段存在是因为首次实现通常需要 2-3 个修订周期，这是健康的。

## 前提条件

- [ ] 第 3 阶段质量门控通过（所有任务已 QA）
- [ ] 收到第 3 阶段交接包
- [ ] 所有功能已实现并单独验证

## 关键心态

> **现实检查器的默认 verdict 是需要工作。**
> 
> 这不是悲观主义 — 这是现实主义。生产就绪性需要：
> - 完整的用户旅程端到端工作
> - 跨设备一致性（桌面、平板、移动）
> - 负载下的性能（不仅仅是愉快路径）
> - 安全验证（不仅仅是"我们添加了认证"）
> - 规格合规性（每个要求，不是大多数）
>
> 第一次通过获得 B/B+ 评级是正常和预期的。

## 代理激活序列

### 步骤 1：证据收集（第 1-2 天，全部并行）

#### 📸 证据收集器 — 全面视觉证据
```
激活证据收集器进行 [项目] 的全面系统证据收集。

所需交付物：
1. 完整截图套件：
   - 桌面 (1920x1080) — 每个页面/视图
   - 平板 (768x1024) — 每个页面/视图
   - 移动 (375x667) — 每个页面/视图
2. 交互证据：
   - 导航流程（点击前后）
   - 表单交互（空、填充、提交、错误状态）
   - 模态/对话框交互
   - 手风琴/可展开内容
3. 主题证据：
   - 浅色模式 — 所有页面
   - 深色模式 — 所有页面
   - 系统偏好检测
4. 错误状态证据：
   - 404 页面
   - 表单验证错误
   - 网络错误处理
   - 空状态

格式：带有 test-results.json 的截图证据包
时间线：2 天
```

#### 🔌 API 测试器 — 完整 API 回归
```
激活 API 测试器进行 [项目] 的完整 API 回归。

所需交付物：
1. 端点回归套件：
   - 所有端点测试（GET、POST、PUT、DELETE）
   - 认证/授权验证
   - 输入验证测试
   - 错误响应验证
2. 集成测试：
   - 跨服务通信
   - 数据库操作验证
   - 外部 API 集成
3. 边缘情况测试：
   - 速率限制行为
   - 大负载处理
   - 并发请求处理
   - 格式错误的输入处理

格式：每个端点通过/失败的 API 测试报告
时间线：2 天
```

#### ⚡ 性能基准测试器 — 负载测试
```
激活性能基准测试器进行 [项目] 的负载测试。

所需交付物：
1. 10 倍预期流量的负载测试：
   - 响应时间分布（P50、P95、P99）
   - 负载下的吞吐量
   - 负载下的错误率
   - 资源利用率（CPU、内存、网络）
2. 核心 Web 指标测量：
   - LCP（最大内容绘制）< 2.5s
   - FID（首次输入延迟）< 100ms
   - CLS（累积布局偏移）< 0.1
3. 数据库性能：
   - 查询执行时间
   - 连接池利用率
   - 索引有效性
4. 压力测试结果：
   - 断点识别
   - 优雅降级行为
   - 过载后恢复时间

格式：性能认证报告
时间线：2 天
```

#### ⚖️ 法律合规检查器 — 最终合规审计
```
激活法律合规检查器进行 [项目] 的最终合规审计。

所需交付物：
1. 隐私合规验证：
   - 隐私政策准确性
   - 同意管理功能
   - 数据主体权利实现
   - Cookie 同意实现
2. 安全合规：
   - 数据加密（静态和传输中）
   - 认证安全
   - 输入净化
   - OWASP Top 10 检查
3. 监管合规：
   - GDPR 要求（如适用）
   - CCPA 要求（如适用）
   - 行业特定要求
4. 可访问性合规：
   - WCAG 2.1 AA 验证
   - 屏幕阅读器兼容性
   - 键盘导航

格式：合规认证报告
时间线：2 天
```

### 步骤 2：分析（第 3-4 天，并行，步骤 1 之后）

#### 📊 测试结果分析器 — 质量指标聚合
```
激活测试结果分析器进行 [项目] 的质量指标聚合。

输入：所有步骤 1 报告
所需交付物：
1. 聚合质量仪表板：
   - 整体质量评分
   - 类别细分（视觉、功能、性能、安全、合规）
   - 问题严重程度分布
   - 趋势分析（如果多个测试周期）
2. 问题优先级：
   - 关键问题（生产前必须修复）
   - 高问题（生产前应该修复）
   - 中等问题（在下一个冲刺中修复）
   - 低问题（待办事项）
3. 风险评估：
   - 生产就绪概率
   - 剩余风险领域
   - 建议的缓解措施

格式：质量指标仪表板
时间线：1 天
```

#### 🔄 工作流优化器 — 流程效率审查
```
激活工作流优化器进行 [项目] 的流程效率审查。

输入：第 3 阶段执行数据 + 步骤 1 发现
所需交付物：
1. 流程效率分析：
   - Dev↔QA 循环效率（首次通过率，平均重试次数）
   - 瓶颈识别
   - 不同问题类型的解决时间
2. 改进建议：
   - 第 6 阶段操作的流程变更
   - 自动化机会
   - 质量改进建议

格式：优化建议报告
时间线：1 天
```

#### 🏗️ 基础设施维护者 — 生产就绪检查
```
激活基础设施维护者进行 [项目] 的生产就绪性检查。

所需交付物：
1. 生产环境验证：
   - 所有服务健康并响应
   - 自动扩展配置并测试
   - 负载均衡器配置验证
   - SSL/TLS 证书有效
2. 监控验证：
   - 所有关键指标被收集
   - 警报规则配置并测试
   - 仪表板访问验证
   - 日志聚合工作
3. 灾难恢复验证：
   - 备份系统运行
   - 恢复程序记录并测试
   - 故障转移机制验证
4. 安全验证：
   - 防火墙规则审查
   - 访问控制验证
   - 密钥管理确认
   - 漏洞扫描干净

格式：基础设施就绪报告
时间线：1 天
```

### 步骤 3：最终判断（第 5-7 天，顺序）

#### 🔍 现实检查器 — 最终 VERDICT
```
激活现实检查器进行 [项目] 的最终集成测试。

强制流程 — 不要跳过：

步骤 1：现实检查命令
- 验证实际构建的内容（ls，grep 声明的功能）
- 交叉检查声明的功能与规格
- 运行全面的截图捕获
- 审查步骤 1 和步骤 2 的所有证据

步骤 2：QA 交叉验证
- 审查证据收集器发现
- 与 API 测试器结果交叉引用
- 验证性能基准测试器数据
- 确认法律合规检查器发现

步骤 3：端到端系统验证
- 测试完整的用户旅程（不是单个功能）
- 验证所有设备的响应行为
- 端到端检查交互流程
- 审查实际性能数据

步骤 4：规格现实检查
- 引用原始规格的确切文本
- 与实际实现证据比较
- 记录规格与现实之间的每个差距
- 无假设 — 仅证据

VERDICT 选项：
- 就绪：生产就绪的压倒性证据（首次通过罕见）
- 需要工作：已识别具体问题，带有修复列表（预期）
- 未就绪：需要第 1/2 阶段重新访问的主要架构问题

格式：基于现实的集成报告
默认：需要工作，除非另有证明
```

## 质量门控 — 最终门控

| # | 标准 | 阈值 | 所需证据 |
|---|-------|-------|----------|
| 1 | 用户旅程完整 | 所有关键路径端到端工作 | 现实检查器截图 |
| 2 | 跨设备一致性 | 桌面 + 平板 + 移动都工作 | 响应式截图 |
| 3 | 性能认证 | P95 < 200ms，LCP < 2.5s，正常运行时间 > 99.9% | 性能基准测试器报告 |
| 4 | 安全验证 | 零关键漏洞 | 安全扫描 + 合规报告 |
| 5 | 合规认证 | 满足所有监管要求 | 法律合规检查器报告 |
| 6 | 规格合规 | 100% 的规格要求已实现 | 逐点验证 |
| 7 | 基础设施就绪 | 生产环境已验证 | 基础设施维护者报告 |

## 门控决策

**唯一权威**：现实检查器

### 如果就绪（进入第 5 阶段）：
```markdown
## 第 4 阶段 → 第 5 阶段交接包

### 给发布团队：
- 现实检查器认证报告
- 性能认证
- 合规认证
- 基础设施就绪报告
- 已知限制（如果有）

### 给增长黑客：
- 产品已准备好面向用户
- 营销信息的功能列表
- 可信度的性能数据

### 给 DevOps 自动化师：
- 生产部署已批准
- 蓝绿部署计划
- 回滚程序已确认
```

### 如果需要工作（返回第 3 阶段）：
```markdown
## 第 4 阶段 → 第 3 阶段返回包

### 修复列表（来自现实检查器）：
1. [关键问题 1]：[描述 + 证据 + 修复说明]
2. [关键问题 2]：[描述 + 证据 + 修复说明]
3. [高问题 1]：[描述 + 证据 + 修复说明]
...

### 流程：
- 问题进入 Dev↔QA 循环（第 3 阶段机制）
- 每个修复必须通过证据收集器 QA
- 所有修复完成后 → 返回第 4 阶段步骤 3
- 现实检查器用更新的证据重新评估

### 预期：2-3 个修订周期是正常的
```

### 如果未就绪（返回第 1/2 阶段）：
```markdown
## 第 4 阶段 → 第 1/2 阶段返回包

### 已识别的架构问题：
1. [基本问题]：[为什么不能在第 3 阶段修复]
2. [结构问题]：[在架构级别需要更改的内容]

### 推荐行动：
- [ ] 修改系统架构（第 1 阶段）
- [ ] 重建基础（第 2 阶段）
- [ ] 缩小范围并重新定义（第 1 阶段）

### 需要工作室制作人决策
```

---

*当现实检查器发出带有压倒性证据的就绪 verdict 时，第 4 阶段完成。需要工作是预期的首次通过结果 — 这意味着系统正在工作但需要润色。*

# 🚨 运行手册：事件响应

> **模式**：NEXUS-Micro | **持续时间**：几分钟到几小时 | **代理**：3-8个

---

## 场景

生产环境中出现问题。用户受到影响。响应速度很重要，但正确处理也同样重要。本运行手册涵盖从检测到事后分析的全过程。

## 严重性分类

| 级别 | 定义 | 示例 | 响应时间 |
|-------|-----------|----------|--------------|
| **P0 — 严重** | 服务完全中断，数据丢失，安全漏洞 | 数据库损坏，DDoS攻击，认证系统故障 | 立即（全员响应） |
| **P1 — 高** | 主要功能故障，显著性能下降 | 支付处理中断，50%+错误率，10倍延迟 | < 1小时 |
| **P2 — 中等** | 次要功能故障，有可用的解决方法 | 搜索不工作，非关键API错误 | < 4小时 |
| **P3 — 低** | 外观问题，轻微不便 | 样式错误，拼写错误，轻微UI故障 | 下一 sprint |

## 按严重性划分的响应团队

### P0 — 严重响应团队
| 代理 | 角色 | 行动 |
|-------|------|--------|
| **基础设施维护者** | 事件指挥官 | 评估范围，协调响应 |
| **DevOps 自动化专家** | 部署/回滚 | 必要时执行回滚 |
| **后端架构师** | 根本原因调查 | 诊断系统问题 |
| **前端开发者** | UI端调查 | 诊断客户端问题 |
| **支持响应者** | 用户沟通 | 状态页面更新，用户通知 |
| **执行摘要生成器** | 利益相关者沟通 | 实时执行更新 |

### P1 — 高响应团队
| 代理 | 角色 |
|-------|------|
| **基础设施维护者** | 事件指挥官 |
| **DevOps 自动化专家** | 部署支持 |
| **相关开发者代理** | 修复实施 |
| **支持响应者** | 用户沟通 |

### P2 — 中等响应
| 代理 | 角色 |
|-------|------|
| **相关开发者代理** | 修复实施 |
| **证据收集器** | 验证修复 |

### P3 — 低响应
| 代理 | 角色 |
|-------|------|
| **Sprint 优先级排序器** | 添加到待办事项 |

## 事件响应序列

### 步骤1：检测与分类（0-5分钟）

```
触发：监控警报 / 用户报告 / 代理检测

基础设施维护者：
1. 确认警报
2. 评估范围和影响
   - 有多少用户受到影响？
   - 哪些服务受到影响？
   - 数据是否有风险？
3. 分类严重性（P0/P1/P2/P3）
4. 激活适当的响应团队
5. 创建事件频道/线程

输出：事件分类 + 响应团队已激活
```

### 步骤2：调查（5-30分钟）

```
并行调查：

基础设施维护者：
├── 检查系统指标（CPU、内存、网络、磁盘）
├── 审查错误日志
├── 检查最近的部署
└── 验证外部依赖

后端架构师（如果是P0/P1）：
├── 检查数据库健康状况
├── 审查API错误率
├── 检查服务通信
└── 识别故障组件

DevOps 自动化专家：
├── 审查最近的部署历史
├── 检查CI/CD管道状态
├── 准备必要的回滚
└── 验证基础设施状态

输出：已识别根本原因（或缩小到组件）
```

### 步骤3：缓解（15-60分钟）

```
决策树：

如果由最近的部署引起：
  → DevOps 自动化专家：执行回滚
  → 基础设施维护者：验证恢复
  → 证据收集器：确认修复

如果由基础设施问题引起：
  → 基础设施维护者：扩展/重启/故障转移
  → DevOps 自动化专家：支持基础设施更改
  → 验证恢复

如果由代码错误引起：
  → 相关开发者代理：实施热修复
  → 证据收集器：验证修复
  → DevOps 自动化专家：部署热修复
  → 基础设施维护者：监控恢复

如果由外部依赖引起：
  → 基础设施维护者：激活回退/缓存
  → 支持响应者：向用户沟通
  → 监控外部恢复

全过程：
  → 支持响应者：每15分钟更新状态页面
  → 执行摘要生成器：向利益相关者简要汇报（仅限P0）
```

### 步骤4：解决方案验证（修复后）

```
证据收集器：
1. 验证修复解决了问题
2. 截图工作状态的证据
3. 确认没有引入新问题

基础设施维护者：
1. 验证所有指标恢复正常
2. 确认没有级联故障
3. 修复后监控30分钟

API 测试员（如果与API相关）：
1. 对受影响的端点运行回归测试
2. 验证响应时间正常化
3. 确认错误率处于基线

输出：事件已解决确认
```

### 步骤5：事后分析（48小时内）

```
工作流优化器领导事后分析：

1. 时间线重建
   - 问题何时引入？
   - 何时检测到？
   - 何时解决？
   - 总用户影响持续时间

2. 根本原因分析
   - 什么失败了？
   - 为什么失败？
   - 为什么没有更早发现？
   - 5个为什么分析

3. 影响评估
   - 受影响的用户
   - 收入影响
   - 声誉影响
   - 数据影响

4. 预防措施
   - 什么监控会更早发现这一点？
   - 什么测试可以防止这一点？
   - 需要什么流程变更？
   - 需要什么基础设施变更？

5. 行动项目
   - [行动] → [负责人] → [截止日期]
   - [行动] → [负责人] → [截止日期]
   - [行动] → [负责人] → [截止日期]

输出：事后分析报告 → Sprint 优先级排序器将预防任务添加到待办事项
```

## 沟通模板

### 状态页面更新（支持响应者）
```
[时间戳] — [服务名称] 事件

状态：[调查中 / 已识别 / 监控中 / 已解决]
影响：[用户影响描述]
当前行动：[我们正在做什么]
下次更新：[何时期待下次更新]
```

### 执行更新（执行摘要生成器 — 仅限P0）
```
事件简报 — [时间戳]

情况：[服务] [中断/降级] 影响 [N用户/流量百分比]
原因：[已知/正在调查] — [如果已知，简要描述]
行动：[正在做什么] — 预计时间 [时间估计]
影响：[业务影响 — 收入、用户、声誉]
下次更新：[时间戳]
```

## 升级矩阵

| 条件 | 升级到 | 行动 |
|-----------|------------|--------|
| P0 30分钟未解决 | 工作室制作人 | 额外资源，供应商升级 |
| P1 2小时未解决 | 项目牧羊人 | 资源重新分配 |
| 怀疑数据泄露 | 法律合规检查器 | 监管通知评估 |
| 用户数据受影响 | 法律合规检查器 + 执行摘要生成器 | GDPR/CCPA通知 |
| 收入影响 > $X | 财务跟踪器 + 工作室制作人 | 业务影响评估 |
